Section 01
專案概述與 POC 目標
背景

翔飛達（洗車設備商）現有 RFID 儲值卡系統資料僅存於實體卡，卡損即資料全滅，補發完全靠客人口述，無法稽核。本 POC 目標是驗證「交易資料即時上傳雲端 → 業者後台可查 → 補發有憑有據」這條路徑是否技術可行，並作為對翔飛達展示用的 Demo。

POC 成功標準
  • 模擬一筆儲值 / 消費交易，資料出現在後台
  • 後台可查詢該卡交易紀錄與餘額
  • 補發流程走完五個步驟（查詢→確認→補發→舊卡停用）
不在 POC 範圍內
  • 實體 RFID 硬體整合（模擬即可）
  • 真正的 WiFi IoT 卡（用 API 模擬上傳）
  • 帳號系統 / 登入驗證
  • 多租戶（multi-tenant）架構
  • 生產環境部署與資安強化
Section 02
POC 範圍界定
功能區塊 In / Out 說明
模擬設備上傳 IN 提供一個「模擬終端」頁面，可手動輸入卡號 + 交易類型 + 金額，送出後呼叫後端 API，模擬 IoT 卡上傳動作
後台查詢介面 IN 輸入卡號，列出該卡完整交易紀錄與目前餘額
補發流程 IN 五步驟補發：查詢→身分確認→餘額確認→補發→停用舊卡（全部 UI 流程，後端紀錄狀態）
會員資料綁定 IN 辦卡時填入姓名 / 電話 / 生日，與卡號做綁定（POC 無驗證）
設備管理 簡化 僅需能新增終端名稱 + 類型，不需即時上線監控
帳號登入 OUT POC 無需驗證，直接進入後台
IoT 硬體 OUT 用「模擬上傳」頁面代替
電子發票 / 第三方支付 OUT 不在此次範圍
Section 03
資料結構規格

POC 後端使用 JSON 檔案或 SQLite 儲存即可。以下為各資料表欄位定義。

💳
cards — 儲值卡主表
關聯：members.id, transactions.card_uid
欄位型態必填說明
uidstring✓RFID 卡片唯一識別碼（模擬用隨機字串即可）
member_idstring | null—綁定會員 ID，辦卡時填入
balanceinteger✓目前餘額（以元為單位，整數）
statusenum✓active / suspended（停用後不可再消費）
created_atdatetime✓建立時間
suspended_atdatetime | null—停用時間（補發流程最後一步寫入）
👤
members — 會員主表
關聯：cards.member_id
欄位型態必填說明
idstring✓UUID
namestring✓姓名（補發身分驗證用）
phonestring✓電話（補發身分驗證用）
birthdaydate✓生日（YYYY-MM-DD，補發身分驗證第三因子）
created_atdatetime✓建立時間
📋
transactions — 交易紀錄表
每筆交易 append-only，不修改歷史紀錄
欄位型態必填說明
idstring✓UUID
card_uidstring✓對應 cards.uid
terminal_idstring✓來源終端 ID
typeenum✓topup（儲值）/ consume（消費）/ reissue（補發）
amountinteger✓金額（正整數，type 決定加減）
balance_afterinteger✓交易後餘額（快照，查詢時不需重算）
created_atdatetime✓交易時間戳記
🖥️
terminals — 終端設備表
POC 只需基本識別，不做即時監控
欄位型態必填說明
idstring✓UUID
namestring✓顯示名稱（例：1號入口加值機）
typeenum✓topup_terminal / payment_terminal
locationstring—安裝地點描述
Section 04
API 端點規格
POC 後端使用 Python + FastAPI。資料存 SQLite（搭配 aiosqlite 或直接用同步 sqlite3 標準庫）即可，不需要 PostgreSQL 或 MySQL。所有回應格式為 JSON。建議安裝：fastapi uvicorn[standard] aiosqlite python-multipart。
POST/api/transactions/upload# 模擬 IoT 卡上傳一筆交易
GET /api/cards/{uid}# 查詢卡片狀態與目前餘額
GET /api/cards/{uid}/transactions# 查詢該卡所有交易紀錄
POST/api/cards# 新增卡片（辦卡）
POST/api/members# 新增會員
PUT /api/cards/{uid}/bind# 將卡片綁定至會員
POST/api/cards/{uid}/reissue# 補發流程：驗證身分 + 建新卡 + 停用舊卡
GET /api/terminals# 列出所有終端
POST/api/terminals# 新增終端

補發端點（reissue）接收 JSON body：{ name, phone, birthday, new_card_uid }。後端比對 member 資料，三者全符合才執行補發，否則回傳 HTTP 403。FastAPI 可用 Pydantic BaseModel 定義這些欄位，自動驗證型態。

Section 05
前端模組規格
技術建議：React（Vite）+ React Router。UI 元件庫不限定，Tailwind 或 plain CSS 皆可。不要求視覺美觀，但流程必須走通。FastAPI 後端需加 CORSMiddleware，允許 localhost:5173（Vite dev server）。
📡
模擬終端頁面
route: /simulator
  • 必下拉選擇終端（terminal_id）
  • 必輸入卡號 UID
  • 必選擇交易類型：儲值 / 消費
  • 必輸入金額，送出呼叫 POST /transactions/upload
  • 必顯示交易結果（成功 / 失敗 / 餘額不足）
  • 選顯示目前卡片餘額（送出後即時更新）
💳
辦卡 / 會員綁定
route: /cards/new
  • 必輸入卡號 UID（模擬掃卡）
  • 必勾選「是否登記會員」
  • 必若登記：輸入姓名 / 電話 / 生日
  • 必輸入初始儲值金額
  • 必送出後顯示建卡成功確認
🔍
交易查詢頁面
route: /cards/:uid
  • 必輸入卡號，顯示餘額 + 卡片狀態
  • 必列出完整交易紀錄（時間、類型、金額、交易後餘額）
  • 必顯示綁定會員姓名（若有）
  • 選依日期區間篩選
  • 選顯示停用標記與停用時間
🔄
補發流程頁面
route: /cards/:uid/reissue
  • 必Step 1：輸入舊卡號，查詢是否存在
  • 必Step 2：輸入姓名 / 電話 / 生日驗證身分
  • 必Step 3：顯示查詢到的餘額，業者確認
  • 必Step 4：輸入新卡 UID，送出補發
  • 必Step 5：顯示完成訊息，舊卡已停用
🖥️
終端管理（簡化）
route: /terminals
  • 必列出所有終端（名稱、類型、ID）
  • 必新增終端（名稱 / 類型 / 地點）
  • 選顯示各終端最後一筆交易時間
🏠
後台首頁 / 導航
route: /
  • 必側邊導覽列，連結各功能頁
  • 選首頁顯示總卡數、總交易數（summary 卡片）
  • 選近期交易列表（最新 10 筆）
必 = 必須完成（POC demo 必要） 選 = 時間允許再做

Section 07
技術限制與注意事項
後端注意事項（FastAPI）
  • 補發操作（reissue）必須是原子性操作：用 with conn: 包住所有 SQLite 操作，任何步驟失敗自動 rollback
  • 交易紀錄 append-only，不可修改或刪除歷史紀錄
  • 餘額快照（balance_after）每筆交易都要寫入，不要每次重算
  • 消費時若卡片 status 為 suspended，必須拒絕並 raise HTTPException(status_code=400)
  • FastAPI 自動生成 /docs（Swagger UI），可直接用來測試所有 API，比 Postman 方便
  • Pydantic BaseModel 定義所有 request / response schema，型態錯誤會自動回傳 422
前端注意事項
  • 補發流程為多步驟 wizard，狀態需用 state 管理（React useState 即可），不用 Redux
  • 所有 API 錯誤都要顯示給使用者看，不能 silent fail
  • 卡號 UID 為 string，不要用 parseInt 處理
  • 金額一律以「元」整數顯示，不要出現小數點
交件與驗收標準

驗收時需要現場（或錄影）走完以下完整流程一次：
建立終端 → 辦卡（含會員登記）→ 儲值 → 消費 → 查詢餘額與交易紀錄 → 補發流程（含身分驗證）→ 確認舊卡停用。
所有步驟無錯誤、資料前後一致，即視為通過。
